# ZionHome — Business Rules & System Rulings

This document captures the official business logic rulings for the ZionHome HOA Management System.
These rules must be followed when generating or modifying any financial or billing-related code.

---

## 1. Association Dues — Payment Rules

### 1.1 Structure

```
AssociationDueLookup (billing year config)
    └── AssociationDueLookupItems (12 monthly templates)

AssociationDue (per homeowner per property per year)
    └── AssociationDueItems (12 monthly line items)
```

### 1.2 Overall Status Derivation

The `AssociationDue.association_due_status_id` is **never set manually** after initial creation.
It must always be **recalculated** based on its items after any payment is posted.

| Condition                | Overall Status  |
| ------------------------ | --------------- |
| 0 items paid             | Unpaid          |
| Some items paid, not all | Partial Payment |
| All items paid           | Paid            |
| Manually cancelled       | Cancelled       |

### 1.3 Item Status

`AssociationDueItem.association_due_item_status_id` must only change via a **posted Transaction**.
Direct manual updates to item statuses are not allowed through the UI.

## 2. Transaction Rules

### 2.1 Core Principle

Transactions are the **single source of truth** for all money movement.
No billing record (AssociationDue, CommunityCharge, Expense) can be marked paid without a transaction.

### 2.2 Transaction Statuses

| Code      | Meaning                                               |
| --------- | ----------------------------------------------------- |
| `DRAFT`   | Just created, not yet verified                        |
| `PENDING` | Submitted for verification                            |
| `POSTED`  | Verified and posted — triggers downstream updates     |
| `VOIDED`  | Cancelled after posting — reverses downstream updates |

### 2.3 Posting Rules

When a Transaction is **posted** (`status = POSTED`):

1. `Transaction.posted_at` and `Transaction.posted_by` are set
2. Each linked `TransactionItem.association_due_item_id` → item status set to **Paid**
3. `AssociationDue` overall status is **recalculated** (see §1.2)
4. A **JournalEntry** is auto-generated (see §3)

### 2.4 Voiding Rules

When a Transaction is **voided** (`status = VOIDED`):

1. `Transaction.voided_at` and `Transaction.void_reason` are set
2. Each linked `AssociationDueItem` status is **reverted to Unpaid**
3. `AssociationDue` overall status is **recalculated**
4. The linked `JournalEntry` status is set to **VOIDED**

### 2.5 Partial Payment Flow

When a homeowner pays only some months (e.g. Jan–Mar out of 12):

```
Record Payment action triggered
    → 1 Transaction created (amount = sum of selected months)
    → 3 TransactionItems created (one per selected month)
    → Transaction posted
        → Jan item → Paid
        → Feb item → Paid
        → Mar item → Paid
        → AssociationDue overall → Partial Payment (3 of 12)
        → JournalEntry generated (Dr. Cash / Cr. HOA Dues Revenue)
```

A **separate Transaction** is created for each payment event.
One homeowner may have multiple transactions against the same AssociationDue.

## 3. Journal Entry Rules

### 3.1 Auto-generation

Journal Entries are **never created manually** for transaction-linked entries.
They are auto-generated by `TransactionService::post()`.

### 3.2 Default Accounting Mapping

| Transaction Type            | Debit                  | Credit                  |
| --------------------------- | ---------------------- | ----------------------- |
| Association Dues Collection | Cash on Hand (1010)    | HOA Dues Revenue (4010) |
| Community Charge Collection | Cash on Hand (1010)    | Other Income (4090)     |
| Expense Payment             | Expense Account (5xxx) | Cash on Hand (1010)     |

### 3.3 Journal Entry Statuses

| Status   | Meaning                                    |
| -------- | ------------------------------------------ |
| `DRAFT`  | Not yet posted                             |
| `POSTED` | Auto-set when transaction is posted        |
| `VOIDED` | Auto-set when linked transaction is voided |

### 3.4 Balance Rule

Every Journal Entry **must balance**: `SUM(debit) === SUM(credit)`.
The system must validate this before allowing a JournalEntry status to be `POSTED`.

## 5. Record Payment — UI Flow (from AssociationDue)

The admin can record payment directly from the AssociationDue list or edit page
using the **Record Payment** action button.

```
Admin opens AssociationDue
    → clicks "Record Payment"
    → modal: select months (CheckboxList of unpaid items)
    → fill: payment method, payment date, receipt/proof, notes
    → total amount auto-computed from selected months
    → submit
        → Transaction created (DRAFT)
        → TransactionItems created (one per selected month)
        → TransactionService::post() called immediately
            → Transaction → POSTED
            → Selected items → Paid
            → Overall status → recalculated
            → JournalEntry → auto-generated
    → success notification with Transaction Ref No.
```

The **Record Payment** button is only visible when the AssociationDue
has at least one item with status: `Unpaid`, `Draft`, or `Partial`.

## 6. What Must NOT Happen

| Rule                                                       | Reason                                                  |
| ---------------------------------------------------------- | ------------------------------------------------------- |
| Do NOT mark `association_due_items.status` directly        | Status must only change via a posted Transaction        |
| Do NOT create JournalEntries manually for payments         | They are auto-generated by `TransactionService::post()` |
| Do NOT set `AssociationDue.status` manually after creation | It is derived from item statuses                        |
| Do NOT skip Transaction creation when recording payment    | All money movement must have a transaction record       |
| Do NOT post a Transaction without TransactionItems linked  | Items define what is being paid                         |

---

## 7. TransactionService — Responsibilities

`App\Services\TransactionService`

| Method                      | Responsibility                                                       |
| --------------------------- | -------------------------------------------------------------------- |
| `post(Transaction)`         | Posts transaction, marks items paid, syncs due status, generates JE  |
| `void(Transaction, reason)` | Voids transaction, reverts item statuses, voids JE, syncs due status |

This service is the **only place** that orchestrates the full payment posting flow.
Do not replicate this logic elsewhere.
